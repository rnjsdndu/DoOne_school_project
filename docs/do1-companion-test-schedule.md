## 3. 날짜별 상세 일정

### 1일차 - 2026-04-20(월)
사전 준비 및 환경 세팅

- 스테이징 서버 접속 상태 확인
- DB 초기화 및 기본 세팅 검증
- 일반/관리자 테스트 계정 생성 및 권한 부여
- 테스트 케이스 시트 1차 리뷰 및 항목 조정
- API 연동 상태와 외부 모듈(SMS, 이메일 등) 연결 점검
- 기본 데이터 입출력 및 더미 데이터 준비
- 일일 마감과 2일차 준비 사항 공유

### 2일차 - 2026-04-21(화)
핵심 기능 테스트

- 회원가입, 로그인, 비밀번호 찾기 등 인증 플로우 점검
- 메인 페이지와 GNB 네비게이션 렌더링 정상 여부 확인
- 핵심 비즈니스 로직과 상세 페이지 동작 확인
- 필수값 누락, 비정상 데이터 입력 시 validation 메시지 점검
- 발견된 크리티컬, Blocker 이슈는 우선 개발팀에 인계
- TC 결과 업데이트와 일일 리포트 작성

### 3일차 - 2026-04-22(수)
세부 기능 테스트 및 1차 수정

- 마이페이지, 내 정보 수정, 회원 탈퇴 등 세부 기능 확인
- 게시판, 고객문의, FAQ 등 서브 페이지 노출과 흐름 점검
- 2일차에 리포트된 버그 수정본 재검증
- 어드민 페이지의 기본 기능(회원 관리, 게시물 관리) 확인
- 잔여 버그 리스트 최신화 및 심각도 재분류
- TC 업데이트, 버그 리포트 등록, 이슈 상태 변경

### 4일차 - 2026-04-23(목)
회귀 테스트 및 호환성 점검

- Chrome, Safari, Edge 등 주요 브라우저 크로스 브라우징 점검
- 모바일 뷰포트에서 레이아웃과 터치 영역 동작 확인
- 모바일 이슈와 UI/UX 결함 목록 정리
- 수정 사항 반영 후 사이드 이펙트와 회귀 여부 점검
- 해결되지 않은 버그 중 Must-Fix 항목 선정
- 회귀 테스트 결과와 모바일 이슈 정리본 공유

### 5일차 - 2026-04-24(금)
최종 재검증 및 결과 보고

- Must-Fix로 지정된 항목 최종 수정 반영 여부 확인
- Minor, Trivial 등급 버그를 후속 수정 대상으로 분류
- 테스트 과정에서 생성된 더미 데이터 삭제 및 정리
- 전체 TC PASS 여부와 Known Issues를 최종 점검
- 기획, 개발, PM 등 유관부서에 결과 보고서 공유
- 테스트 종료 선언 및 최종 결과 보고 완료

## 4. 운영 유의사항 및 원칙

- 버그 우선순위는 `Blocker`, `Critical`부터 즉시 개발팀에 공유한다.
- 버그 리포팅 시 `환경 정보`, `재현 단계`, `기대 결과`, `실제 결과`를 반드시 함께 기록한다.
- 스크린샷이나 화면 녹화는 가능한 한 함께 첨부한다.
- 개발팀에서 수정 완료를 전달하더라도 QA 담당자가 직접 재검증해 `PASS` 처리하기 전에는 티켓을 종료하지 않는다.
- 서버 이슈 등으로 일정이 지연되면 PM과 즉시 협의해 회귀 테스트 범위나 인력을 조정한다.

## 5. 최종 결론 및 기대 효과

5일간의 집중 테스트를 통해 DO1 웹사이트의 핵심 기능을 조기에 검증하고, 사용자에게 안정적인 경험을 제공하는 것을 목표로 한다.
명확한 버그 트래킹과 잔존 이슈 관리를 통해 론칭 품질을 확보하며, 최종 결과 보고서로 후속 개선 과제를 정리한다.
